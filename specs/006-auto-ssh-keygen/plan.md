# Implementation Plan: Auto SSH Key Generation During Enrollment

**Branch**: `006-auto-ssh-keygen` | **Date**: 2026-03-22 | **Spec**: [spec.md](spec.md)

## Summary

Replace the "no SSH key found" error in both enrollment paths with automatic key generation. The enrollment script (POSIX shell, generated by the hub) will silently generate an ed25519 key pair. The CLI `enroll` command (Go binary) will prompt the user for confirmation before generating.

## Technical Context

**Language/Version**: Go 1.26.1 (CLI binary) + POSIX shell (enrollment script generated server-side)
**Primary Dependencies**: Standard library, `os/exec` for `ssh-keygen` invocation in Go, `ssh-keygen` CLI tool in shell script
**Storage**: Filesystem (`~/.ssh/id_ed25519`, `~/.ssh/id_ed25519.pub`)
**Testing**: `go test` (standard testing package)
**Target Platform**: Linux and macOS (both enrollment paths)
**Project Type**: CLI tool + web service (hub generates enrollment script)
**Performance Goals**: N/A (one-time operation during enrollment)
**Constraints**: Must use `ssh-keygen` (not a Go crypto library) to generate standard OpenSSH key files; must not overwrite existing keys
**Scale/Scope**: 2 code paths (shell script + Go CLI), 2 files modified

## Design Decisions

### 1. Shell Script vs Go Library for Key Generation

**Decision**: Use `ssh-keygen` CLI tool in both paths (shell script calls it directly; Go calls it via `os/exec`).

**Rationale**: `ssh-keygen` produces standard OpenSSH key files with correct permissions automatically. Using a Go crypto library would require manually writing files with correct formats and permissions — more complex, more error-prone, and unnecessary when the tool is universally available on target platforms.

**Alternatives rejected**:
- Go `crypto/ed25519` + manual file writing: More code, must handle OpenSSH key format encoding, permission setting, and `.pub` file generation manually. No benefit over calling `ssh-keygen`.

### 2. Enrollment Script: Auto-Generate vs Prompt

**Decision**: Auto-generate without prompting in the shell script.

**Rationale**: The enrollment script runs via `curl | sh` — there's no reliable interactive stdin. The user has already expressed intent to enroll by running the script. Generating a key is a prerequisite, not a separate decision.

### 3. CLI Command: Prompt Before Generating

**Decision**: Prompt the user for confirmation in the CLI `enroll` command.

**Rationale**: The CLI user is in an interactive terminal and may have opinions about key paths, types, or whether to use an existing key elsewhere. A quick "y/n" prompt respects user agency without adding friction.

### 4. Key Generation Function Location

**Decision**: Create `internal/agent/keygen.go` with an `EnsureSSHKey()` function.

**Rationale**: Keeps key generation logic testable and separate from enrollment. The function checks if the key exists, and if not, calls `ssh-keygen`. Returns the key path (existing or newly generated).

### 5. Generated Key Path

**Decision**: Always generate at `~/.ssh/id_ed25519` (the default path).

**Rationale**: This is the first path the enrollment script checks, and the default `--key` flag value in the CLI. Using the standard path ensures maximum compatibility with other SSH tools.
